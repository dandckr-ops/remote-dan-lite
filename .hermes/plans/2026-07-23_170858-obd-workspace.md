# Remote Dan Lite OBD Workspace Implementation Plan

> **For Hermes:** Execute this plan with strict RED → GREEN → REFACTOR slices. Preserve the canonical checkout and live appliance runtime until source verification is green.

**Goal:** Add a local-first OBD workspace to Remote Dan Lite with direct OBDLink SX support, simulated operation, live SAE J1979 values, fault reading and governed clearing, VIN/vehicle information, customer and vehicle records, and durable evidence lineage.

**Architecture:** Extend the existing FastAPI process and SQLite evidence database rather than creating a second service or database. Put SAE/ELM decoding and hardware ownership in dedicated `remote_dan.obd_*` modules; routes in `app.py` call an `OBDService`. Existing `assets → cases → sessions → captures → artifacts` lineage remains authoritative. OBD-specific tables reference those records and store query/audit metadata while large/raw evidence remains hashed on disk.

**Tech stack:** Python 3.11+, FastAPI, SQLite, pyserial, static HTML/CSS/JavaScript, pytest, Playwright/Chromium for rendered verification when available.

---

## Safety and scope

- Generic legislated OBD-II only: SAE J1979 Modes `$01`, `$03`, `$07`, `$09`, and `$0A`; Mode `$04` only through a prepared one-time confirmation.
- No coding, flashing, actuator control, security access, enhanced Subaru modules, or manufacturer-specific commands.
- Mode `$04` is one attempt only. Never retry after timeout or ambiguous transport state.
- The clear warning must state that readiness monitors, freeze-frame data, and emissions diagnostic information can be erased/reset; clearing is not a repair.
- The OBDLink SX is separate hardware from the ECOM/CANCapture passive CAN lane.
- One service-owned lock guards the OBD adapter. Browser polling cannot overlap serial transactions.
- The live runtime is `/opt/remote-dan-lite`; source work occurs only in `/home/ares/worktrees/remote-dan-lite-obd` until activation is separately approved.

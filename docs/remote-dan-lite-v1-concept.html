Design notes
Why this shape works

Product boundary

A session-centered evidence appliance. Capture engines, protocol satellites, evidence storage, and the operator UI cooperate without pretending every capability lives in one board or one process.

Why the three-plane split matters

  • If capture hangs, routing still exists.
  • If the router is fine but the Pi wedges, OOB can recover it.
  • The operator does not lose the laptop to the car-side hardware mess.

Target session bundle

One bounded diagnostic session is intended to correlate:

  • Pico waveforms
  • CAN log
  • serial trace
  • battery voltage
  • bounded Modbus transaction log
  • event markers
  • PDF summary + raw files

Internal topology

AC IEC ----\ > power ---- router 12/24V ----/ ├─ capture node └─ OOB node operator ETH --------> router uplink ETH ----------> router router <-------------> capture router <-------------> OOB router <---- LAN ----> Modbus satellite ---- RS-485 target capture ---- metadata -> local SQLite capture ---- artifacts -> local filesystem OOB ---- power-cycle -> capture

Evidence index

SQLite stores metadata and durable lineage: asset → case → session → capture → artifact. Waveform CSV, PNG, PDF, JSON, and future protocol logs remain ordinary files with database IDs, paths, sizes, media types, and SHA-256 checksums.

Modbus boundary

The Anybus gateway is a LAN satellite for structured Modbus TCP-to-RTU transactions. It does not replace the direct isolated serial or scope lane required for raw electrical and timing evidence.

Brand carryover
